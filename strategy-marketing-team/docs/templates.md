# docs/templates.md — 협업 문서 표준 구조

오브온 프로젝트에서 Claude Code와 Codex가 만드는 모든 협업/결과물 문서의 표준 구조다. 파일명은 `docs/collab-protocol.md` §3 `파일 네이밍 규칙`에 따라 slug를 붙인다.

모든 섹션을 반드시 채울 필요는 없다. 비워둘 때는 "해당 없음"이라고 표시한다.

## 인덱스

| 문서 | Owner | 트리거 | 섹션 번호 |
|---|---|---|---|
| `dev_plan_<slug>.md` | Claude Code | 복잡한 개발/분석 작업 전 1차 계획 | 1 |
| `questions_<slug>.md` | Claude Code | 대표 질문 / 분석 프레임 설계 | 2 |
| `risk_notes_<slug>.md` | Claude Code | 데이터/전략 리스크 점검 | 3 |
| `final_report_<slug>.pdf` | Claude Code | 대표용 최종 전략보고서 | 4 |
| `handoff_<slug>.md` (Claude) | Claude Code | Claude Code 세션 인계 | 5 |
| `execution_plan_<slug>.md` | Codex | Claude Code 1차 계획 보완 | 6 |
| `handoff_<slug>.md` (Codex) | Codex | Codex 작업 인계 | 7 |
| `analysis_summary_<slug>.md` | Codex | 데이터 분석 요약 | 8 |

## 1. dev_plan_<slug>.md (Claude Code)

```
# dev_plan_<slug>.md

## Goal
이번 작업으로 무엇을 만들거나 바꿀지 한두 문장.

## Scope
포함할 일.

## Out of Scope
이번 작업에서는 하지 않을 일. 과잉 구현 방지.

## Files
- 수정 예상 파일
- 건드리면 안 되는 파일 (Codex가 작업 중이거나 사용자만 만지는 파일)
- 새로 생성할 파일 (slug 적용 파일명)

## Proposed Steps
Codex가 실행할 순서. 단계별로 짧게.

## Risks
- 데이터 해석 리스크 (샘플/실제 혼동, 추정 위험)
- 파일 충돌 가능성
- 과잉 구현 가능성
- 시각/톤 일관성 리스크

## Verification
Codex가 확인해야 할 기준 (테스트, 렌더링, 파일 검사, 데이터 검증).

## Handoff Needed
- 작업 끝나고 누가 이어받는지
- 어떤 산출물을 누구에게 넘기는지
- 최종 문서화가 필요한지
```

## 2. questions_<slug>.md (Claude Code)

# questions_<slug>.md

## Decision Context
이번에 답해야 할 의사결정 한 줄.

## Key Questions
대표가 결정 전에 답해야 할 질문 3~7개. 각 질문은:
- 어떤 데이터로 답하는가

질문을 어떤 프레임(비교 축, 시계열, 코호트 등)으로 본 것인지.

## 4. final_report_<slug>.pdf (Claude Code)

대표가 5분 안에 의사결정할 수 있게 **핵심 → 근거 → 액션** 순서로 구성한다. `business-context.md` §9 의사결정 순서를 따르되, PDF 형식에 맞는 시각·구조 요소(표지·KPI Snapshot·페이지 구성·차트 placeholder·시각 가이드)를 추가한다.

### 페이지 구성 (총 7~9p 권장)

| # | 페이지 | 분량 | 핵심 콘텐츠 |
|---|---|---|---|
| 1 | 표지 + Executive Summary | 1p | 한 줄 결론 · 핵심 액션 3개 미리보기 |
| 2 | 이번 달 한눈에 (KPI Snapshot) | 1p | 핵심 지표 카드 · 전월 대비 · 한 문장 요약 |
| 3 | 다음 달 밀 상품 | 1~2p | 선정 · 근거 데이터 · 실행 액션 · 측정 KPI · 메시지 톤 |
| 4 | 개선 대상 상품·메시지 | 1p | 무엇/왜(가설)/어떻게 · 검증 방법 · 톤 가이드 |
| 5 | 인스타 콘텐츠 방향 3개 | 1~2p | 방향별 카드 (메시지·사용 장면·활용 지표·콘텐츠 예시·발행 시점) |
| 6 | 리스크 3개 + 대응 | 1p | 영향도 · 트리거 신호 · 대응 액션 · 모니터링 지표 |
| 7 | 액션 플랜 (5개) | 1p | 우선순위 · 담당 · 시점 · KPI · 의존성 |
| 8 | Appendix | 1p | 데이터 출처 · 해석 한계 · 다음 보고 주기 · 참고 차트 |

각 페이지 끝에 `---`(수평선) 또는 `<div style="page-break-after: always;"></div>`로 page break 표시. Codex가 PDF 변환 시 실제 페이지 분할 처리.

### 시각·서식 가이드

- **색상 · 여백 · 폰트**: `DESIGN.md` 토큰 따름.
- **차트 제목은 해석형**: "매출 추이" ❌ / "선물세트 매출 높지만 환불도 높음" ✅
- **표 · 차트 placeholder**: 본문에서 `[차트 1. ...]`, `[표 1. ...]` 형식으로 명시. Codex가 PDF 변환 시 실제 시각 자료로 교체.
- **페이지 break**: 각 `## ` 헤딩 앞에 `---`(수평선). 강제 break 필요 시 `<div style="page-break-after: always;"></div>`.
- **폰트 권장** (PDF 변환 도구에 전달): 본문 11pt · 헤딩 14~18pt · 캡션 9pt.
- **분량 통제**: 표지 + Executive Summary와 KPI Snapshot은 각 1페이지 안에 들어가게.
- **인스타 지표**: `AGENTS.md` §10 "사용 가능 지표"만 사용.
- **단정 금지**: 가설은 조건부로 (예: "환불 사이클 + 시즌 효과로 추정"). 단정형 결론 금지.
