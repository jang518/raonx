# PROJECT.md

경영지원팀 CAE 매출 대시보드 프로젝트의 현황판. Codex와 Claude Code가 협업할 때 보는 단일 진실 원천이다.

**Goal**: 경영지원팀에서 Excel로 관리 중인 CAE 소프트웨어 판매 / 유지보수 / 용역 / 컨설팅 등 매출 데이터를 사내 웹 페이지 대시보드로 확인할 수 있게 한다. 최종 목표는 경영진과 경영지원팀이 팀별, 기간별, 손익 항목별 실적과 목표 진행률을 빠르게 파악하고, 반복적인 Excel 수작업 확인을 줄이는 것이다.

**Primary users**: 경영진, 경영지원팀.

**운영 환경**: 회사 내 24시간 가동 중인 Windows 11 PC를 서버 역할로 사용한다. 대시보드는 사내 접근용 웹 페이지로 운영하며, 외부 공개를 전제로 하지 않는다.

**배포 방향**: Docker Compose 기반 배포로 진행한다. Windows 11 PC에서 Docker Desktop + WSL 2 기반으로 실행하고, 앱 서버 / 데이터 처리 / 필요 시 DB 또는 캐시를 컨테이너 단위로 분리한다. Docker Compose 구성에는 앱 서비스, 데이터 볼륨, 환경 변수 파일, 헬스체크, 재시작 정책, 운영 문서를 포함한다.

**메타 규칙**: `docs/collab-protocol.md` — 역할, Work Board/락/Decisions/Handoff Log 갱신 방식, 파일 네이밍, 협업 환경 정책, archive 규칙.

## Decisions

* 사용자 요청이 기본 협업 모델보다 우선한다.
* actor 간 이견은 사용자가 직접 우선순위와 담당자를 지정해서 해결한다.
* 복잡한 작업은 Claude Code 계획 → Codex 보완·구현 → Claude Code 최종 문서화 흐름.
* 협업 문서는 이어받기 필요한 작업에서만 만든다.
* 모든 협업/결과물 문서는 작업별 slug를 붙여 구분한다.
* 파일 동시 수정은 Work Board `in_progress` 상태와 `Files locked` 컬럼으로 락을 잡아 방지한다.
* 협업 모델은 옵션 B (PROJECT.md 중앙 락, 메인 SSOT). actor 4명+ 또는 동시 작업 빈번 시 재검토.
* Codex 기본은 메인 직접 작업, 큰 작업/사용자 명시 지시 시 채팅 시작 전 "Start in → New worktree" 선택.
* Claude Code 기본은 백그라운드 세션 시 워크트리 자동, 라이브 세션은 메인 직접.
* PROJECT.md / 메타 문서 / git 설정 파일은 양쪽 모두 메인에서 직접 수정한다.
* Codex 워크트리는 채팅 시작 전 "Start in → New worktree"로만 시작. 작업 중간 "Handoff to worktree" 금지 (메인 reset 위험).
* 경영지원 데이터 작업은 기본적으로 단일 순차 작업으로 진행한다. 병렬 작업은 사용자 명시 지시가 있을 때만 예외적으로 허용한다.
* 원천 Excel 파일은 직접 덮어쓰지 않는다. 원천 데이터는 `data/raw/`에 보관하고, 정제 데이터는 `data/processed/`, DB, 또는 캐시 파일로 분리한다.
* 원천 Excel, 정제 데이터, DB 파일, 로그 파일, `.env`, 인증 정보, 사내 IP/포트 정보는 `.gitignore` 대상으로 관리한다.
* CAE 매출 데이터는 회사 내부 정보로 취급한다. 고객사명, 계약 금액, 유지보수 금액, 용역/컨설팅 매출, 팀별 실적, 담당자 정보 등은 외부 공개 금지로 간주한다.
* 대시보드는 경영진과 경영지원팀만 사용하는 사내 조회 화면을 우선한다.
* 대시보드는 로그인 가능하도록 만든다. 초기 인증 방식은 구현 난이도와 보안 수준을 비교한 뒤 결정하되, 최소한 사용자별 로그인과 세션 관리를 포함한다.
* 사용자 권한은 초기에는 `admin`, `executive`, `business_support` 정도의 단순 역할 기반 접근 제어로 시작한다.
* 외부 인터넷 공개, 공개 GitHub 업로드, 실제 데이터가 포함된 샘플 공유, 외부 클라우드 배포는 사용자 승인 없이는 진행하지 않는다.
* 서버 역할은 회사 내 24시간 가동 Windows 11 PC가 담당한다.
* Windows 11 운영 PC 배포는 Docker Compose 기반으로 진행한다.
* Docker Compose 구성에는 최소한 앱 서비스, 데이터 볼륨, 환경 변수 파일, 헬스체크, 재시작 정책을 포함한다.
* Docker 배포 파일은 `deploy/compose.yaml`, `.env.example`, `deploy/README.md`를 기준으로 만들고, 운영자가 실행/중지/재시작/백업 방법을 알 수 있게 한다.
* 첫 구현 목표는 완성도 높은 대규모 시스템이 아니라, Excel 기반 매출 데이터를 웹에서 안전하게 조회할 수 있는 최소 기능 대시보드다.
* 데이터 입력/수정 기능은 초기 범위에서 제외한다. Excel 원천 데이터를 기준으로 불러오기, 정제, 집계, 시각화하는 흐름을 먼저 만든다.
* 초기 대시보드는 읽기 전용 조회 화면으로 만든다.
* Excel 파일 교체 또는 지정 폴더 업로드 후 재집계하는 방식으로 데이터 업데이트를 처리한다.
* 자동 새로고침이 필요한 경우, 먼저 수동 업로드/수동 재집계 기능을 안정화한 뒤 주기적 재집계를 검토한다.
* 첨부된 현재 Excel 기준 핵심 원천 시트는 `팀별실적`과 `목표`다.
* `팀별실적` 시트는 팀별 월별 실적과 손익 항목의 핵심 원천으로 사용한다.
* `목표` 시트는 목표값 비교, 목표 달성률, 목표 대비 진행률, 팀별 기여도 계산에 사용한다.
* `실적G` 시트는 샘플 대시보드 작성 과정에서 사용한 임시 FCST 시트로 취급한다. 향후 업데이트 시 없을 가능성이 높으므로 초기 대시보드 계산 범위에서는 제외하고, 실적 데이터 기반 대시보드를 우선한다.
* `total`, `실적(23~26)`, `경영현황비교` 시트는 참고용 또는 향후 확장 후보로 둔다. 초기 메인 대시보드 계산은 `팀별실적` + `목표` 중심으로 진행한다.
* 팀별 개별 시트(`SDT`, `TST1`, `TST2`, `MKT`, `EST1`, `EST2`, `EST3` 등)는 상세 검증 또는 향후 drill-down 후보로 둔다.
* 대시보드의 기본 분석 단위는 팀별, 월별, 분기별 누적, 반기별 누적, 연간 누적이다.
* 매출 구분은 현재 Excel 구조에서는 손익 항목 중심으로 처리한다. CAE 사업 유형인 소프트웨어 판매 / 유지보수 / 용역 / 컨설팅 구분은 원천 Excel에 명확한 컬럼 또는 매핑 규칙이 확인된 뒤 추가한다.
* `팀별실적` 시트 기준으로 우선 사용하는 항목은 `매출`, `매입`, `매출이익`, `영업이익`, `경상이익`, `직접 비용`, `간접 비용`, `경비`다.
* `경비` 항목이 명시적으로 있으면 그대로 사용하고, 없으면 `직접 비용 + 간접 비용`으로 계산한다.
* 월 컬럼은 `1월`부터 `12월`까지를 기본으로 사용한다.
* 분기 컬럼은 `Q1`, `Q2`, `Q3`, `Q4`를 기본으로 사용한다.
* 연간 합계 컬럼은 해당 연도 컬럼을 사용하되, 연도명이 바뀔 수 있으므로 설정 파일에서 지정 가능하게 한다.
* 대시보드 상단 KPI는 전역 필터 기준으로 `총매출`, `매출이익`, `영업이익`, `경비`, `전월 대비 영업이익 증감률`, `전월 대비 경비 증감률`을 우선 표시한다.
* 차트는 `매출 추이`, `영업이익 추이`, `매출이익 추이`, `팀별 매출 비교`, `팀별 매출이익 비교`, `팀별 영업이익 비교`, `팀별 매출 기여도`, `목표 대비 매출이익 진행률`, `목표 대비 팀 기여도`, `연간 목표 달성률`을 우선 표시한다.
* 하단 상세 테이블은 `팀명`, `매출`, `매출이익`, `영업이익`, `경비`, `목표대비매출이익기여도`, `전체진행률기여도`, `연간 목표달성률`, `연간 목표대비부족액`을 우선 표시한다.
* 현재 구조상 월 목표가 없고 분기/반기/연간 목표 중심이므로, 월별 조회 시 목표 관련 일부 지표는 현재 분기 목표 기준 fallback으로 계산한다.
* 목표 fallback 규칙은 코드에 하드코딩하지 않고 `config/dashboard.yml` 또는 동등한 설정 파일로 분리한다.
* 대시보드 항목, KPI, 차트, 테이블 컬럼, 목표 fallback, 팀 그룹, 시트명, 컬럼명, 연도 컬럼은 코드 수정 없이 바꿀 수 있도록 설정 파일로 분리한다.
* 설정 파일 후보는 `config/dashboard.yml`, `config/data-mapping.yml`, `config/metrics.yml`로 둔다.
* `config/data-mapping.yml`은 Excel 시트명, 컬럼명, 행/열 매핑, 팀명 표준화 규칙을 관리한다.
* `config/metrics.yml`은 매출, 매입, 매출이익, 영업이익, 경비, 목표달성률, 기여도 등 계산식을 관리한다.
* `config/dashboard.yml`은 화면에 표시할 KPI, 차트, 테이블, 필터, 정렬 기준을 관리한다.
* 설정 파일 변경 시에는 대시보드 항목이 깨지지 않도록 스키마 검증 또는 설정 검증 스크립트를 둔다.
* 대시보드 항목을 추가/삭제/변경할 때는 먼저 설정 파일을 수정하고, 필요한 경우에만 앱 코드를 수정한다.
* 설정 파일에는 실제 고객사명, 비밀번호, 사내 IP, 계정 정보 등 민감 정보를 넣지 않는다.
* 배포 환경별 값은 `.env` 또는 Docker Compose 환경 변수로 분리한다.
* `.env` 파일은 커밋하지 않고, `.env.example`만 커밋한다.
* 매출 인식 기준, 부가세 포함 여부, 유지보수 기간 배분, 다년 계약 처리 방식은 사용자 확인 전까지 임의로 확정하지 않는다.
* 금액 집계는 원천 Excel에 있는 값과 명시된 계산 규칙만 사용한다.
* 계약 가능성, 예상 매출, 영업 확률, 미확정 수주, 고객사별 전망은 사용자가 명시적으로 제공하지 않는 한 추정하지 않는다.
* FCST는 초기 대시보드 계산 범위에서 제외한다.
* `실적G`가 있더라도 샘플/임시 데이터로 취급하고, 사용자 결정 없이 실적을 보완하거나 덮어쓰는 데 사용하지 않는다.
* 향후 FCST를 정식 기능으로 사용할 경우에는 표시 방식, 계산 기준, 신뢰 가능 시점, 반영 대상 항목을 별도 결정한 뒤 설정 파일에 명시한다.
* 데이터 검증 결과는 숨기지 않는다. 누락 컬럼, 빈 금액, 날짜/월 형식 오류, 중복 팀명, 음수 매출, 비정상적으로 큰 금액, Excel 수식 오류는 별도 경고로 표시한다.
* 현재 첨부 Excel에서 일부 참고 시트에 수식 오류로 보이는 값이 있을 수 있으므로, 메인 계산에 쓰는 시트와 참고 시트를 구분하고 데이터 품질 리포트를 남긴다.
* 원천 Excel의 병합 셀, 빈 셀, 반복 생략된 구분값은 정제 단계에서 명시적으로 채운다.
* 팀별 데이터는 원천 표 형태를 그대로 화면에 노출하지 않고, 정규화된 테이블 구조로 변환한 뒤 대시보드에서 사용한다.
* 정제 데이터의 기본 구조는 `year`, `period_type`, `period`, `team`, `metric`, `amount`, `source_sheet`, `source_cell`, `updated_at` 형태를 우선 검토한다.
* 로그인 기록, 데이터 업로드 기록, 재집계 기록은 운영 로그로 남긴다.
* 운영 PC에는 정기 백업 규칙을 둔다. 최소한 원천 Excel, 설정 파일, DB/캐시 파일, 배포 설정은 백업 대상이다.
* PROJECT.md는 현황판(Work Board + Decisions + Handoff Log) 3섹션으로만 운영하고, 상세 메타 규칙은 `docs/collab-protocol.md`로 분리한다.
* 최종 산출물은 사내 웹 대시보드, Excel 데이터 정제 스크립트, 설정 파일, 로그인 기능, Docker Compose 배포 파일, 운영 방법 문서, 백업/복구 가이드로 구성한다.

## Handoff Log

작업 종료 시 `[YYYY-MM-DD HH:mm] actor / changed: ... / verified: ... / next: ...` 형식으로 1줄 기록한다. 최근 10개를 유지하고 초과분은 `archive/`로 이동한다(상세 `docs/collab-protocol.md` §6).

*(기록 없음)*
