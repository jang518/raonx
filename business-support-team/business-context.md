# business-context.md — 경영현황 AI 대시보드 사업 맥락

> 목적: Claude Code와 Codex가 경영지원팀의 CAE 매출/손익 데이터를 해석할 때 같은 업무 기준을 보게 하기 위한 문서입니다.
> 이 문서는 기술 구현 문서가 아니라, 숫자를 어떻게 해석하고 어떤 판단을 하면 안 되는지 정하는 기준 문서입니다.

## 1. 프로젝트 개요

프로젝트 이름: `management-ai-dashboard`
표시명: 경영현황 AI 대시보드 프로젝트
대시보드 이름: 경영현황 대시보드

이 프로젝트는 경영팀이 AI를 현업에 적용하기 위한 첫 번째 실무형 프로젝트다. 현재 경영지원팀에서 Excel로 관리하고 있는 CAE 소프트웨어 판매, 유지보수, 용역, 컨설팅 관련 매출/손익 데이터를 사내 웹 대시보드로 확인할 수 있게 만드는 것이 목표다.

대시보드를 만드는 가장 큰 이유는 다음과 같다.

* 대시보드 생성에 들어가는 리소스 줄이기
* 데이터 생성 담당자가 아니어도 내용을 쉽게 파악할 수 있게 하기
* 경영회의 때 현황을 빠르게 공유하기
* 정보 공유용 자료 작성 시간을 줄이기
* Excel 기반 수작업 분석과 반복 보고 부담을 줄이기

현재 Excel 관리 방식의 주요 불편은 다음과 같다.

* 대시보드와 보고 자료를 만들기 위해 리소스가 많이 든다.
* 데이터 생성 담당자 외에는 내용을 파악하기 어렵다.
* 경영회의용 자료를 별도로 정리하는 시간이 필요하다.
* 숫자의 의미와 계산 기준이 파일을 만든 사람에게 의존하기 쉽다.

## 2. 최종 사용자

주요 사용자는 다음과 같다.

| 사용자      | 사용 목적                                    |
| -------- | ---------------------------------------- |
| 경영진      | 경영회의에서 팀별 실적, 목표 달성률, 손익 흐름을 빠르게 확인      |
| 경영지원팀    | Excel 기반 데이터를 정리하고, 경영진 공유용 자료 작성 시간을 줄임 |
| 프로젝트 담당자 | AI를 활용한 경영 현황 공유 자동화 가능성 검증              |

초기 대시보드는 경영진과 경영지원팀이 같은 화면을 보는 구조로 시작한다. 필요하면 이후 사용자 역할별 화면 분리 또는 권한 분리를 검토한다.

## 3. 회사/사업 맥락

회사는 CAE 해석을 위한 Pre-/Post software와 PIDO 관련 소프트웨어를 판매하고, 이와 관련된 기술 지원, 기술 용역, 컨설팅, 국책 과제, PoC를 수행한다.

주요 사업/매출 유형은 다음과 같다.

| 구분      | 의미                                                  |
| ------- | --------------------------------------------------- |
| 신규 라이선스 | 신규 고객 또는 기존 고객의 추가 소프트웨어 구매로 발생하는 매출                |
| 갱신      | 기존 라이선스 계약을 갱신하면서 발생하는 매출                           |
| 유지보수    | 라이선스를 매년 갱신할 때 발생하는 비용                              |
| 기술 용역   | 단순 소프트웨어 구매가 아니라, 고객사가 발주한 CAE 관련 프로젝트 수행으로 발생하는 매출 |
| 기술 지원   | 고객사의 CAE 엔지니어링 업무를 지원하면서 발생하는 업무                    |
| 컨설팅     | 기술 자문 및 세미나 발표로 발생하는 매출                             |
| 국책 과제   | 정부/공공 과제 참여로 발생하는 매출 또는 과제성 수익                      |
| PoC     | 고객사 또는 프로젝트의 가능성 검증을 위한 개념검증성 업무                    |

현재 Excel 구조에서는 사업 유형별 구분보다 팀별 실적과 목표 중심으로 데이터가 정리되어 있다. 따라서 초기 대시보드는 `팀별실적`과 `목표` 시트를 중심으로 구현한다. 소프트웨어 판매 / 유지보수 / 용역 / 컨설팅 구분은 원천 Excel에 명확한 컬럼 또는 매핑 규칙이 확인된 뒤 확장한다.

## 4. 주요 고객/시장

주요 고객은 조선, 자동차, 항공, 전자 등 제조업체의 CAE 엔지니어링 업무 담당자다.

대시보드에서 고객사별 상세 분석은 초기 범위에서 필수로 두지 않는다. 초기 대시보드는 팀별, 기간별, 손익 항목별 실적과 목표 달성률을 확인하는 데 집중한다.

## 5. 조직/팀 구분

현재 Excel 데이터는 팀별로 구분되어 있다.

| 팀     | 의미 / 주요 역할                  |
| ----- | --------------------------- |
| SDT   | 프로그래밍을 하는 개발팀               |
| TST1  | 국내 영업팀                      |
| TST2  | 해외 영업팀                      |
| MKT   | 마케팅팀                        |
| EST1  | 기술 용역 및 기술 지원팀              |
| EST2  | 기술 용역팀                      |
| EST3  | 기술 지원팀                      |
| 경영지원팀 | 경영 현황 관리, 자료 정리, 대시보드 운영 지원 |

팀별 실적을 비교할 때 일반적으로 별도 주의사항은 없지만, **기술지원팀은 단순 매출만으로 평가하면 안 된다.** 기술지원팀은 직접 매출보다 고객 유지, 기술 대응, 유지보수 안정성, 용역/컨설팅 지원 같은 간접 기여가 있을 수 있다.

팀별 목표는 모두 같은 기준으로 잡혀 있지 않다. 따라서 팀별 목표 달성률을 비교할 때는 단순 순위보다 각 팀의 역할과 목표 기준 차이를 함께 고려해야 한다.

## 6. 매출/손익 항목 해석 기준

회사가 매출을 볼 때 중요하게 보는 항목은 다음과 같다.

| 항목              | 중요도   | 해석 기준                         |
| --------------- | ----- | ----------------------------- |
| 매출              | 중요    | 전체 사업 규모와 팀별 외형 성과 확인         |
| 매출이익            | 중요    | 매출에서 매입 등 직접 원가를 제외한 이익성 확인   |
| 영업이익            | 중요    | 실제 운영 관점에서의 수익성 확인            |
| 경상이익            | 중요    | 비영업 요소까지 포함한 경상적 손익 확인        |
| 경비              | 중요    | 매출/이익 대비 비용 증가 여부 확인          |
| 목표 달성률          | 가장 중요 | 경영회의에서 가장 핵심적으로 확인할 지표        |
| 미래 기술 트렌드 사전 준비 | 중요    | 당장의 매출 외에 향후 준비가 필요한 기술 영역 확인 |

대시보드에서는 단순히 매출이 큰 팀을 보여주는 데서 끝나지 않고, 목표 달성률, 매출이익, 영업이익, 경비 흐름을 함께 보여줘야 한다.

## 7. 이번 프로젝트 목표

이번 프로젝트의 목표는 Excel 기반 경영현황 데이터를 사내 웹 대시보드로 전환해 경영회의와 내부 공유에 사용할 수 있게 하는 것이다.

대시보드가 답해야 하는 핵심 질문은 다음과 같다.

1. 현재 기간의 총매출, 매출이익, 영업이익, 경비는 얼마인가?
2. 팀별 매출과 매출이익 기여도는 어떻게 다른가?
3. 팀별 목표 달성률은 어느 수준인가?
4. 목표 대비 부족액이 큰 팀은 어디인가?
5. 전월 대비 영업이익과 경비는 어떻게 변했는가?
6. 매출은 증가했지만 이익성은 악화된 팀이 있는가?
7. 경비 증가가 영업이익에 영향을 주는 구간이 있는가?
8. 데이터 누락, 수식 오류, 이상치가 있는가?
9. 경영회의에서 바로 공유할 수 있는 핵심 포인트는 무엇인가?

## 8. 사용 데이터

초기 데이터는 사용자가 제공한 Excel 파일을 기준으로 한다.

초기 핵심 원천 시트:

| 시트     | 역할                            |
| ------ | ----------------------------- |
| `팀별실적` | 팀별 월별 실적과 손익 항목의 핵심 원천        |
| `목표`   | 팀별 목표값, 목표 달성률, 목표 대비 진행률 계산용 |

선택 또는 참고 시트:

| 시트          | 역할                                      |
| ----------- | --------------------------------------- |
| `실적G`       | 샘플로 만든 임시 FCST 시트. 향후 업데이트 시 없을 가능성이 높음 |
| `total`     | 현재 메인 계산 원천은 아니며 참고용                    |
| `실적(23~26)` | 현재 메인 계산 원천은 아니며 참고용 또는 향후 확장 후보        |
| `경영현황비교`    | 참고용 또는 향후 확장 후보                         |
| 팀별 개별 시트    | 상세 검증 또는 향후 drill-down 후보               |

초기 메인 대시보드 계산은 `팀별실적` + `목표` 중심으로 진행한다.

## 9. Excel 항목 기준

`팀별실적` 시트 기준으로 우선 사용하는 항목은 다음과 같다.

* `매출`
* `매입`
* `매출이익`
* `영업이익`
* `경상이익`
* `직접 비용`
* `간접 비용`
* `경비`
* `1월 ~ 12월`
* 월별, 분기별, 반기별, 연간 집계 구조

`경비` 항목이 명시적으로 있으면 그대로 사용한다. `경비`가 없으면 `직접 비용 + 간접 비용`으로 계산한다.

대시보드 기본 연도는 **2026년**으로 한다.

목표 시트의 목표는 현재 기준으로 **분기별 팀당 매출 목표 금액**으로 해석한다. 다만 사용자는 목표를 연간/분기/월 기준으로 관리할 수 있다고 했으므로, 구현 시 목표 기준은 고정값으로 하드코딩하지 않고 설정 파일에서 조정 가능하게 둔다.

## 10. 목표 해석 규칙

목표 달성률은 이 프로젝트에서 가장 중요한 지표다.

목표 관련 기본 원칙은 다음과 같다.

* 목표는 팀별로 서로 다른 기준을 가질 수 있다.
* 목표 비교는 단순 순위가 아니라 팀별 역할과 목표 기준 차이를 고려해야 한다.
* 목표값이 명확히 있는 기간은 해당 목표를 사용한다.
* 월 목표가 없는 경우 월 목표가 있는 것처럼 표시하지 않는다.
* 월 목표가 없는데 월별 조회를 할 경우, 목표 관련 지표는 `없음`, `목표 없음`, 또는 `분기 목표 기준 참고값`으로 표시한다.
* 월 목표 fallback을 사용할 경우 화면에 반드시 표시한다.

권장 표시 방식:

| 상황                  | 표시 방식                   |
| ------------------- | ----------------------- |
| 월 목표가 있음            | 월 목표 대비 달성률 표시          |
| 월 목표가 없음            | `월 목표 없음` 표시            |
| 월별 화면에서 목표 참고값이 필요함 | `현재 분기 목표 기준 참고값`으로 표시  |
| 목표 기준이 불명확함         | 목표 관련 지표를 계산하지 않고 경고 표시 |

월 목표가 없을 때 임의로 월 목표를 1/3 배분하거나 선형 배분하는 방식은 초기 기본값으로 사용하지 않는다. 필요하면 사용자 결정 후 설정 파일에 명시한다.

## 11. FCST / 실적G 해석 규칙

`실적G` 시트는 이번 샘플 대시보드를 만들기 위해 임시로 만든 데이터다. 향후 업데이트 시 없을 가능성이 높다.

따라서 기본 원칙은 다음과 같다.

* `실적G` 시트가 없으면 FCST는 사용하지 않는다.
* `실적G` 시트가 있어도 확정 운영 기준으로 보지 않는다.
* FCST 관련 기능은 초기 대시보드의 필수 기능이 아니다.
* 실적과 FCST가 동시에 있으면 실적을 우선한다.
* FCST가 반영된 값은 화면에서 별도 표시하지 않는 것을 기본으로 한다.
* 다만 향후 FCST를 정식으로 사용할 경우에는 표시 방식과 계산 기준을 다시 정한다.

현재 기준에서는 FCST 관련 로직을 최소화하고, 실적 데이터 기반 대시보드를 우선한다.

## 13. 의사결정 기준

### 좋은 흐름으로 볼 수 있는 경우

다음 조건을 함께 만족하면 긍정적인 흐름으로 본다.

* 매출과 매출이익이 함께 증가한다.
* 목표 달성률이 계획 대비 양호하다.
* 경비 증가율보다 영업이익 증가율이 높다.
* 팀별 매출 기여도와 목표 기여도가 모두 높다.
* 연간 목표대비부족액이 줄어든다.

### 확인이 필요한 경우

다음 조건이면 경영진 또는 경영지원팀 확인이 필요하다.

* 매출은 증가했지만 영업이익이 감소한다.
* 매출이익은 증가했지만 경비도 크게 증가한다.
* 특정 팀의 매출 기여도는 높지만 목표 달성률은 낮다.
* 기술지원팀처럼 단순 매출만으로 평가하기 어려운 팀이 낮게 보인다.
* 연간 목표대비부족액이 큰 팀이 있다.
* 전월 대비 증감률이 비정상적으로 크다.
* 월별 수치가 비정상적으로 튀거나 음수/빈 값이 있다.
* Excel 수식 오류 또는 누락 컬럼이 있다.

### 판단을 보류해야 하는 경우

다음 조건이면 결론을 내리지 않고 확인 필요로 표시한다.

* 원천 Excel에 필요한 컬럼이 없다.
* 목표 기준이 명확하지 않다.
* 월 목표가 없는데 월 목표 달성률을 요구한다.
* `실적G`가 없는데 FCST 기반 해석을 요구한다.
* 사업 유형별 매출 구분 컬럼이 없는데 소프트웨어 판매/유지보수/용역/컨설팅별 분석을 요구한다.
* 매출 인식 기준, 부가세 포함 여부, 다년 계약 처리 기준이 확인되지 않았다.

## 14. 데이터 해석 금지 사항

다음은 Claude Code와 Codex 모두 지켜야 한다.

* 원천 Excel에 없는 값을 추정하지 않는다.
* 계약 가능성, 예상 매출, 영업 확률, 미확정 수주는 사용자가 제공하지 않는 한 추정하지 않는다.
* 매출 인식 기준을 임의로 확정하지 않는다.
* 부가세 포함/제외 여부를 임의로 판단하지 않는다.
* 유지보수 기간 배분과 다년 계약 처리를 임의로 계산하지 않는다.
* 월 목표가 없는데 월 목표가 있는 것처럼 표현하지 않는다.
* `실적G`가 없는데 FCST가 있는 것처럼 계산하지 않는다.
* 기술지원팀을 단순 매출 기준만으로 부정적으로 평가하지 않는다.
* 데이터 품질 경고를 숨기지 않는다.
* 실제 고객사명, 계약 금액, 담당자 정보, 로그인 정보, 사내 IP/포트를 외부 공개하지 않는다.

## 15. 설정 기반 관리 원칙

향후 대시보드 항목을 쉽게 수정할 수 있도록, 화면 구성과 계산 기준은 코드에 직접 박아 넣지 않고 설정 파일로 분리한다.

우선 설정 파일 후보는 다음과 같다.

| 파일                        | 역할                               |
| ------------------------- | -------------------------------- |
| `config/data-mapping.yml` | Excel 시트명, 컬럼명, 팀명 매핑            |
| `config/metrics.yml`      | 매출, 매출이익, 영업이익, 경비, 목표 달성률 등 계산식 |
| `config/dashboard.yml`    | KPI, 차트, 테이블, 필터, 정렬 기준          |

설정 파일로 관리할 항목:

* KPI 표시 여부
* 차트 표시 여부
* 테이블 컬럼
* 팀명 표준화
* 목표 기준
* 기본 연도
* 기본 기간
* 금액 단위
* 목표 fallback 방식

설정 파일에는 실제 고객사명, 비밀번호, 사내 IP, 계정 정보 등 민감 정보를 넣지 않는다.

## 16. 보안 및 접근 기준

대시보드는 사내 사용을 전제로 한다.

기본 원칙:

* 로그인 기능을 포함한다.
* 초기 사용자는 경영진, 경영지원팀, 프로젝트 담당자로 둔다.
* 외부 공개를 전제로 하지 않는다.
* 실제 데이터가 들어간 파일은 공개 저장소에 올리지 않는다.
* `.env`, 비밀번호, 세션 키, 사내 IP/포트 정보는 커밋하지 않는다.
* 사용자 계정과 권한은 초기에는 단순한 역할 기반으로 시작한다.

초기 권한 후보:

| 역할                 | 설명                        |
| ------------------ | ------------------------- |
| `admin`            | 계정/설정/데이터 업데이트 관리         |
| `executive`        | 경영현황 대시보드 조회              |
| `business_support` | 데이터 확인, 경영회의 자료 확인, 운영 지원 |

## 17. 배포 및 운영 기준

배포는 Docker 기반으로 확정한다.

운영 환경:

* 회사 내 24시간 가동 중인 Windows 11 PC
* Docker Desktop + Docker Compose 기반 배포
* 사내 접근용 웹 페이지
* 외부 공개 없음

Docker 배포 시 포함해야 할 항목:

* `deploy/compose.yaml`
* `.env.example`
* `deploy/README.md`
* 앱 서비스
* 데이터 볼륨
* 환경 변수 설정
* 헬스체크
* 재시작 정책
* 백업/복구 안내
* 실행/중지/재시작 방법

운영 초기에는 복잡한 자동화보다 안정적인 수동 실행과 명확한 운영 문서를 우선한다.

## 19. 톤과 문장 기준

경영진과 경영지원팀이 빠르게 이해할 수 있게 쓴다.

좋은 문장:

* “3분기 누적 매출은 증가했지만 경비 증가율이 더 빨라 영업이익률 확인이 필요합니다.”
* “TST1은 매출 기여도는 높지만 목표 대비 진행률은 낮아 원인 확인이 필요합니다.”
* “기술지원팀은 단순 매출 기준만으로 평가하기보다 유지보수 안정성과 기술 대응 기여를 함께 봐야 합니다.”
* “월 목표가 없기 때문에 이 화면의 목표 진행률은 분기 목표 기준 참고값입니다.”
* “`실적G` 시트가 없어 FCST 보완은 적용하지 않았습니다.”

피해야 할 문장:

* “목표 달성은 문제없어 보입니다.”
* “다음 달 매출은 증가할 것입니다.”
* “이 팀은 실적이 나쁩니다.”
* “월 목표 달성률이 낮습니다.”
* “FCST 기준으로 보면 좋아질 것입니다.”

원칙:

* 숫자 근거 없는 단정형 결론 금지
* 추정과 사실을 구분
* 목표 기준과 기간 기준을 명확히 표시
* 경고와 한계를 숨기지 않음
* 경영회의에서 바로 말할 수 있는 문장으로 작성
